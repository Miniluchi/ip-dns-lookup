2. **Helper function** `idleCard<T>(): CardState<T>` — returns `{ status: 'idle', data: null, error: null }`

3. **`useLookup` hook** following the exact pattern from 02-RESEARCH.md Pattern 2:
   - State: `LookupState` (4 cards, all start as `idleCard()`)
   - State: `inputType: InputType` (starts `'unknown'`)
   - State: `isLoading: boolean` (starts `false`)
   - `lookup(raw: string)` callback:
     a. Call `sanitizeInput(raw)` then `detectInputType(sanitized)` — import from `@/lib/detect-input-type`
     b. If sanitized is empty or type is `'unknown'`, return `{ valid: false }` immediately
     c. Set all visible cards to `{ status: 'loading', data: null, error: null }`; rdns stays `idleCard()` when type is `'domain'`
     d. Set `isLoading = true`
     e. Use `fetchCard<T>(url)` helper: fetches URL, parses JSON, returns `CardState<T>` with `status: 'success'` or `status: 'error'` (checking `!res.ok || json.error`)
     f. `Promise.allSettled([geo, dns, rdns, whois])` — rdns resolves to `idleCard()` for domain input
     g. Map settled results: `fulfilled` -> use value, `rejected` -> `{ status: 'error', data: null, error: 'Request failed' }`
     h. `setIsLoading(false)` ALWAYS runs (use try/finally)
     i. Return `{ valid: true }`
   - Returns `{ state, inputType, isLoading, lookup }`

4. **Create `src/hooks/use-lookup.test.ts`** using vitest (globals: true means no imports needed for describe/it/expect):

   - Since vitest environment is `node` (no jsdom), the simplest approach is to test the orchestration logic by extracting a pure async function `performLookup(query: string)` that contains the fetch + Promise.allSettled logic, separate from React state. Export it for testing. The hook wraps this function with useState.
   - Alternative: add `@testing-library/react` to devDependencies and use `renderHook` — if this path is chosen, also update vitest.config.ts to use `environment: 'jsdom'` for files matching `*.test.ts` in hooks directory

**Important:** The `fetchCard` error check must handle both `!res.ok` (HTTP error) AND `json.error` (API returns 200-range but with error body — this doesn't happen in current routes but is defensive).
